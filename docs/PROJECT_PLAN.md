# 项目流程与阶段计划

本文件把项目拆成可以检查的阶段。一个阶段只有在“完成证据”齐全后才能标记完成。

## 1. 每项工作的统一流程

无论是代码、文档还是提问，都按下面的顺序处理：

1. **说明目标**：写清要解决的问题、负责人和完成标准。
2. **确认边界**：指出会改哪些文件，不会碰哪些内容。
3. **建立分支**：从最新 `main` 建立自己的工作分支。
4. **完成工作**：只修改与本任务有关的内容。
5. **自行检查**：按任务标准实际操作，保存命令和结果。
6. **写工作记录**：工作类 PR 同时提交 Markdown 和 JSON 报告。
7. **发起 PR**：目标分支必须是 `main`，使用模板说明结果和风险。
8. **负责人审阅**：仓库拥有者可以提问、要求修改、接受或拒绝。
9. **合并后确认**：只在 PR 合并后，才把任务标记为正式完成。

提问型 PR 不需要测试或审计报告，只要把问题、已尝试内容和希望得到的决定写清楚。

## 2. Phase 0：仓库初始化与约定冻结

目标：所有人从同一套说明开始，不再依赖聊天记录猜测职责。

| 项目 | 负责人 | 交付物 | 完成证据 |
| --- | --- | --- | --- |
| 项目入口和基线 | 仓库拥有者 | README、项目基线 | 新人可从首页找到全部规则 |
| 人员职责 | 1 号、2 号确认 | 人员职责文档 | 没有重复负责或无人负责的核心模块 |
| 接口初版 | 两人共同 | `API_CONTRACT.md` | 请求、响应、枚举和失败行为明确 |
| 提交方式 | 仓库拥有者 | 协作流程、PR 指南和模板 | 无代码经验者可在网页提交提问 PR |
| 合规入口 | 两人共同 | 合规规则、素材来源表 | 每项素材都能记录来源和许可 |

完成门槛：上述文件合并到 `main`，两人阅读并在 PR 中确认。

## 3. Phase 1：两人分别做出最小演示

### 1 号的最小演示

- Windows 桌面出现透明、无边框、可拖动的桌宠窗口。
- 可以加载合法授权的占位 Live2D 模型。
- 可以用 Mock 数据显示回复、播放音频并切换一种表情和动作。
- 服务不可用时显示提示，不导致客户端崩溃。

完成证据：启动步骤、短演示记录、Mock 测试结果、已知问题清单。

### 2 号的最小演示

- 后端可本地启动，`/health` 返回正常状态。
- `/api/chat` 接收文字并返回符合约定的固定 Mock 响应。
- LLM、TTS 或 ASR 尚未接入时，明确标为 Mock，不冒充真实结果。
- 日志能够指出请求成功或失败，但不记录密钥和完整敏感音频。

完成证据：启动步骤、示例请求与响应、测试命令、测试结果。

Phase 1 完成门槛：两人各自的最小演示不依赖对方也能复现。

## 4. Phase 2：第一次端到端联调

联调顺序：

1. 两人再次核对 `/api/chat` 和 `emotion`、`motion` 枚举。
2. 2 号先保证后端能稳定返回文字和 Mock 音频信息。
3. 1 号接入后端地址，验证文字回复。
4. 再接入 TTS 音频播放。
5. 最后验证表情、动作和错误处理。

一次完整成功演示必须出现：

```text
用户输入一句话
→ 后端生成或模拟回复
→ 客户端显示文字
→ 客户端播放语音
→ Live2D 切换约定的表情和动作
```

完成证据：双方使用同一版接口合同、端到端测试记录、失败场景记录、未完成项。

## 5. Phase 3：真实服务与四服务器部署

计划部署顺序：

1. VM-1：API Gateway + Chat Orchestrator。
2. VM-2：LLM Adapter + Session Service，或根据资源情况并入 VM-1。
3. VM-3：TTS Service。
4. VM-4：ASR Service + 静态音频服务或备用服务。

两份来源文档对服务器职责有两种拆法。2 号必须先根据实际资源做一次轻量压力验证，再在部署 PR 中确定最终拓扑；在此之前以上仅为计划，不是已验证架构。

必须验证的失败场景：

- LLM 失败：返回明确的固定文字，不让客户端崩溃。
- TTS 失败：仍返回文字，音频字段为空。
- ASR 失败：提示改用文字输入。
- 网络断开：客户端提示并可重试。
- 未知表情或动作：回到安全默认值。

完成证据：每台服务器的部署步骤、健康检查、资源使用、失败切换结果和回滚说明。不得在报告中记录服务器凭据。

## 6. Phase 4：打包与最终验收

最终验收由两人共同完成，仓库拥有者确认：

- [ ] Windows 客户端可按 README 启动。
- [ ] 桌宠可拖动、置顶、透明显示。
- [ ] Live2D 模型可加载，缺失时有清楚提示。
- [ ] 文字输入能够获得回复。
- [ ] 语音输入能够获得回复，失败时可改用文字。
- [ ] TTS 音频能播放，失败时仍显示文字。
- [ ] `emotion` 和 `motion` 能驱动客户端表现。
- [ ] 网络断开后可提示并重试。
- [ ] 四服务器方案有实际部署或明确的未验证说明。
- [ ] 新人可以只看文档完成基本启动和操作。
- [ ] 素材、声音、代码许可证和来源记录完整。
- [ ] 两条工作线和最终集成都有 Markdown、JSON 两种审计报告。

## 7. 每日同步

建议每天两次，每次只回答固定问题。

上午：

1. 昨天完成了什么？
2. 今天准备做什么？
3. 是否被其他任务阻塞？
4. 是否准备修改共同约定？

晚上：

1. 今天的 PR 或工作分支是什么？
2. 做了哪些实际检查，结果是什么？
3. 还有哪些未完成或有风险？
4. 明天第一件事是什么？

“没有测试”也是允许的答案，但必须写明原因，不能写成“已通过”。
