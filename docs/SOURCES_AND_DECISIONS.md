# 资料来源与整理决定

本文件说明仓库规则来自哪些材料，以及遇到冲突时采用了哪一种方案。原始文件未复制进仓库；仓库保存的是便于执行的整理结果。

## 1. 使用的材料

- `Arknights VTuber Project Plan.pdf`：27 页总体目标、四人三线程方案、阶段、风险、验收和审计要求。
- `GitHub_Collaboration_Workflow_Guide.pdf`：Fork、分支和 Pull Request 到 `main` 的流程。
- `1号人员计划书.txt`：桌宠客户端、Live2D、交互、通信、Mock、打包和验收。
- `2号人员计划书.txt`：后端入口、`/api/chat`、LLM、ASR、TTS、指令生成、部署和验收。

## 2. 人员方案

差异：总体计划按四人拆分，两个独立计划按两人各约 50 点重新合并职责。

当前决定：以更具体的两个独立计划作为当前执行版本；四人方案只在未来扩编时使用。原因是两人计划提供了清楚的相互边界，并且与当前提供的人员文件一一对应。

## 3. 分支方案

差异：总体计划使用 `develop` 和多条长期线程分支；GitHub 协作指南要求所有 PR 最终指向 `main`。

当前决定：仓库规模和参与者经验有限，采用最简单的“短期任务分支 → PR 到 `main`”。不建立长期 `develop` 分支，减少同步和选错目标分支的风险。

## 4. 接口字段

差异：早期总计划使用 `text`，两份人员计划的响应示例使用 `reply_text`；表情和动作名称也有两套。

当前决定：采用两份人员计划中更具体的 `reply_text` 和七项表情、七项动作枚举，记录在 [`API_CONTRACT.md`](API_CONTRACT.md)。早期名称不作为 v0.1 正式值。

## 5. 服务器拓扑

差异：总体计划将 VM-2 作为 TTS、VM-3 作为 ASR、VM-4 作为备用；2 号计划提出 VM-2 运行 LLM Adapter、VM-3 运行 TTS、VM-4 运行 ASR 和静态音频。

当前决定：仓库只记录两种方案存在，不在没有实际资源测试时假装已经选定。2 号在部署阶段通过资源检查和失败测试提出最终拓扑 PR。

## 6. 仍需确认

- `Open-LLM-VTuber` 使用 submodule、源码副本还是外部依赖。
- 项目代码许可证。
- 正式 Live2D、背景和声音的授权。
- 四台服务器的实际可用资源与网络条件。
- 最终技术栈、真实启动命令和测试命令。

这些项目影响实现或发布时，必须通过单独 PR 记录决定。
