# 人员职责

当前使用两人方案。每项职责都包含“要做什么、要交什么、怎样算完成”。

## 1. 1 号：桌宠客户端与角色表现

一句话说明：1 号负责用户在电脑上看见和操作的全部体验。

### A. 桌宠窗口

要做：

- 建立 Windows 桌面客户端。
- 支持透明背景、无边框、置顶和鼠标拖动。
- 提供托盘菜单，以及退出、重启和打开设置入口。
- 程序异常时留下不含敏感信息的基础日志。

完成标志：新机器按说明启动后能看见桌宠，窗口可拖动，托盘操作正常。

### B. Live2D 表现

要做：

- 加载模型和模型配置。
- 播放待机动作、切换表情、控制口型。
- 建立角色状态：`idle`、`thinking`、`speaking`、`happy`、`confused`、`error`。
- 模型缺失或加载失败时显示清楚提示。

完成标志：Mock 响应中的 `emotion` 和 `motion` 能产生可见变化，未知值能安全回退。

### C. 用户操作

要做：

- 文字输入、语音按钮、角色点击、右键菜单和设置面板。
- 显示简易对话历史、错误说明和重试按钮。
- 保存服务地址、模型路径、置顶、缩放、音量、透明度和语音开关。

完成标志：用户能发送文字、看到回复、播放返回音频，并在服务不可用时继续操作设置或重试。

### D. 客户端通信

要做：

- 按 [`API_CONTRACT.md`](API_CONTRACT.md) 调用 `/api/chat`。
- 处理超时、重试、音频播放和联调日志。
- 在 2 号服务未完成时提供 Mock，保证客户端可以独立演示。

完成标志：同一套客户端既能连接 Mock，也能连接 2 号后端；错误不会导致客户端崩溃。

### E. 打包和说明

要做：

- 提供开发启动、构建和 Windows 打包命令。
- 提供客户端 README、配置示例、模型替换方法和演示步骤。

交付物至少包括：客户端代码、Mock、配置模板、启动说明、Windows 包或可复现的打包结果、自测报告。

## 2. 2 号：对话、语音与部署服务

一句话说明：2 号负责接收客户端请求，并稳定返回文字、语音、表情和动作指令。

### A. 后端入口

要做：

- 整理 `Open-LLM-VTuber` 可复用的后端能力。
- 建立统一服务入口、配置读取、日志和 `/health`。
- 提供 `.env.example`，真实密钥只放本地或服务器安全配置中。

完成标志：新人能按后端 README 在本地启动服务，健康检查正常，错误能从日志定位。

### B. `/api/chat`

要做：

- 检查请求字段，管理 `session_id`，读取 `character_id`。
- 文字输入直接进入对话流程；语音输入先进入 ASR。
- 统一返回 `reply_text`、`emotion`、`motion` 和音频信息。
- 用清楚的错误码和说明表示失败。

完成标志：文字和语音两类测试都返回符合接口合同的结构；异常响应也能被 1 号识别。

### C. LLM 对话

要做：

- 加载角色 Prompt，管理基本多轮上下文和回复长度。
- 支持通过配置切换 LLM 提供方。
- 过滤不安全内容，不冒充官方角色或未经确认的官方设定。
- LLM 失败时返回固定的降级回复。

完成标志：回复简短、适合语音播放；切换 Mock 或真实提供方不需要改公共接口。

### D. ASR 和 TTS

要做：

- 封装语音识别和语音合成。
- 统一音频格式、缓存和过期清理。
- 开发环境可返回 Base64；部署环境优先返回 URL。
- TTS 失败仍返回文字；ASR 失败提示用户改用文字。

完成标志：1 号客户端可播放生成音频，缓存不会无限增长，失败行为符合合同。

### E. 表情和动作指令

要做：

- 根据回复生成稳定的 `emotion` 和 `motion`。
- 初期可使用简单规则，后续再替换，不提前增加复杂分类器。
- 只能返回接口合同列出的值。

完成标志：每次成功回复都有两个字段；无法判断时使用安全默认值。

### F. 四服务器部署

要做：

- 提供 Docker、反向代理、环境变量模板、健康检查和部署 README。
- 根据真实资源验证最终服务拆分，不在 1 GiB 服务器上强行运行重型模型。
- 记录启动、重启、日志查看、备份和回滚步骤。

交付物至少包括：后端代码、接口文档、测试脚本、部署配置、健康检查、压测或资源检查结果、后端自测报告。

## 3. 两人共同负责

| 共同事项 | 1 号负责提供 | 2 号负责提供 | 完成标志 |
| --- | --- | --- | --- |
| `/api/chat` | 客户端真实需求和解析结果 | 服务端请求与响应实现 | 同一示例在两边都能运行 |
| 表情与动作 | 模型实际支持的表现 | 稳定的字段值 | 不出现无法识别的值 |
| 错误处理 | 用户看得懂的提示和重试 | 错误码、降级响应和日志 | 任一服务失败时客户端不崩溃 |
| Demo | 客户端操作和画面 | 后端服务和测试数据 | 完成一轮可重复演示 |
| 最终验收 | Windows 包和操作说明 | 部署结果和服务说明 | 验收清单有实际证据 |

任何一方准备修改公共接口时，必须先发 PR 说明原因、影响和兼容办法，另一方确认后再合并。

## 4. 仓库拥有者

仓库拥有者负责：

- 保持 `main` 只包含已经审阅的内容。
- 为 PR 指定或承担审阅责任。
- 在 PR 中回答问题，并明确“接受、需要修改或暂不采用”。
- 处理职责冲突和优先级，不替参与者编造测试结果。
- 确认合规、敏感信息和许可证风险后再公开发布。

## 5. 工作量参考

两份独立计划均估算约 50 点。点数只用于保持工作量大致平衡，不等于小时，也不应用来代替完成证据。

## 6. 明确的工作边界

- 1 号不负责部署后端服务器；2 号不负责客户端窗口和 Live2D 渲染。
- 客户端播放音频由 1 号负责；服务端生成或提供音频由 2 号负责。
- 1 号提供 Mock 用于客户端独立演示；2 号提供 Mock 用于后端无密钥测试。
- 正式素材和声音必须先完成来源登记，任何人都不能因为“只是测试”而提交未授权内容。

## 7. 未来扩充为四人时

如果项目确认增加到四人，可按早期总计划拆分：

| 新分工 | 从当前职责中拆出 |
| --- | --- |
| 人员 1：桌宠客户端与 Live2D | 当前 1 号的窗口、渲染、通信和打包 |
| 人员 2：角色包与 Prompt | 当前 1 号的角色资产，加上当前 2 号的角色 Prompt 规则 |
| 人员 3：后端编排与 LLM | 当前 2 号的后端入口、对话接口、会话和 LLM |
| 人员 4：语音与部署 | 当前 2 号的 ASR、TTS、Docker 和服务器运维 |

扩编前必须更新项目基线和负责人，不能让两套人员编号同时生效。
